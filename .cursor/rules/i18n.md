# i18n Conventions

## Setup

- Library: `next-intl` v4
- Locales: `en` (LTR, default), `he` (RTL)
- Message files: `src/messages/en.json`, `src/messages/he.json`
- Locale config: `src/lib/i18n.js`
- Club language stored in `Club.language` field (enum: `en` | `he`)

## Rules

1. **Always add keys to BOTH `en.json` and `he.json`** — missing keys fall back to English then to the raw key string
2. **Use namespaced keys** — top-level keys are namespaces (e.g., `common`, `auth`, `nav`, `dashboard`, `activities`, `email`, `profile`, `register`, `payment`)
3. **Never hardcode user-facing strings** — always use `useTranslations("namespace")` in client components
4. **Placeholders use `{name}` syntax** in next-intl: `t("greeting", { name: "John" })`

## Client-Side Usage

```js
import { useTranslations } from "next-intl";
const t = useTranslations("namespace");
// then: t("key") or t("key", { param: value })
```

## Email Templates (Server-Side)

`src/lib/email.js` has its own `t()` function that reads from the `email` namespace:
- Uses `{placeholder}` replacement (regex-based), NOT next-intl
- Always use the `email` namespace for email-related strings
- Template HTML is inline in `email.js` — not in separate files

## RTL Support

- `getDirection(locale)` returns `"rtl"` for Hebrew, `"ltr"` for English
- Dashboard layout sets `document.documentElement.dir` based on club language
- Email templates use inline `dir` attribute
- Global CSS has `[dir="rtl"]` overrides for inputs and phone groups

## Adding a New Locale

1. Create `src/messages/{locale}.json` with all namespaces
2. Add locale to `locales` array in `src/lib/i18n.js`
3. Add to `Club.language` enum in `src/models/Club.js`
4. Add direction mapping in `getDirection()`
5. Add date locale mapping in `getDateLocale()`
